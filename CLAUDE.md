## 协作规则

### 防幻觉铁律（CRITICAL — 任何 prompt 模板变更前必读）

LLM 遇到它不知道的数据时不会报错，而是静默编造。三道防线：

1. **注入覆盖**：prompt 中出现的所有股票代码，其市值/PE/PB/名称等客观数据必须通过 `%%PLACEHOLDER%%` 注入。LLM 训练数据里的市值是过时的，必须用实时 API 覆盖。
2. **代码全量提取**：`_extract_codes_from_feeds` 必须从所有注入源（recap/review/zsxq/wechat/news/industry）提取代码，不能遗漏任何 feed。
3. **输出校验**：LLM 输出中的股票名称-代码匹配、大小盘分类、估值区间等可验证字段，Python 端须用行情 API 逐条校验，不匹配的自动修正。

> 历史事故：2026-05-29 prompt 含「读取文件」指令→LLM 编造成交额(8000亿 vs 实际29682亿)、2026-06-03 STOCK_CONTEXT 未覆盖公众号标的→协创数据1100亿被归为小盘股。根因相同：LLM 需要的数据没注入，它就编。

### 名称→代码双通道提取（CRITICAL — 任何从文本提取股票代码的地方必读）

信息源同时存在两种写法：写代码的（「688167」）和只写名称的（「炬光科技」）。只用 `\b\d{6}\b` 正则 = 名称全漏。

**强制规则**：所有从文本提取股票代码的地方，必须走 `data.extract_codes_from_text(text)`（正则6位代码 + 全名反向匹配双通道），不得直接用 `re.findall(r"\b(\d{6})\b", text)`。

**特别注意**：大文件经 Haiku 摘要后会丢失名称。补扫原始文件时只做名称→代码匹配（不做正则），防止代码数从 60 暴涨到 800+。

> 历史事故：2026-06-15 炬光科技(688167)在 zsxq 被提及 12 次（IP 授权台积电 CPO 核心卡位），但全管线认为它不存在。根因：星球帖子只写「炬光科技」不写「688167」，全项目 7 文件 14 处只用正则抓代码。修复：`data.py` 新增名称→代码反向索引，全项目替换为双通道提取。

### 输入审计铁律（CRITICAL — 任何 LLM 分析管线上线/变更后必读）

代码逻辑正确 ≠ LLM 真的收到了正确的数据。LLM 不会告诉你它只拿到了标题——它会用专业的措辞和完整的结构掩盖输入缺失。三道防线：

1. **Prompt 抽查**：新管线上线后，必须打印一次 LLM 实际收到的完整 prompt，人肉确认每个字段都有真实数据。`ann_full_text = title` 这种「看起来有值」的字段骗过了三条管线。
2. **数字信号检验**：LLM 输出中出现具体数字（金额、占比、倍数）是正文生效的强信号。如果输出全是定性描述没有数字，大概率输入端只有标题/摘要——可作为自动化健康检查指标。
3. **共享基础设施优先**：多条管线同时出现同类问题时，先怀疑共享层（PDF→文本、API→字段），不要逐条管线修。三条管线根因相同：缺 PDF 提取层。

> 历史事故：2026-06-14 审计发现 announcement_deep_read/research_deep_read/_analyze_industry 三条管线，LLM 实际只读到标题+元数据。根因：东财 API 只返回元数据，正文在 PDF 里，系统从未做 PDF→文本提取。`ann_full_text = a.get("title", "")` 把标题当全文喂给 LLM。修复：新建 pdf_utils.py（pdfplumber 提取），验证后 Haiku 提取从全 None→具体数字。

### 降级链完整性（CRITICAL — 任何多级降级设计前必读）

降级链的每一级都需要参数，**每个调用点必须传齐所有降级路径需要的参数**。缺一个参数 = 那一级静默失败 = 降级链断裂。三道防线：

1. **调用点审查**：新增降级参数时，grep 所有调用点确认参数已传递。`download_report_pdf(pdf_url, info_code)` 加了 `info_code` 后，`_analyze_industry` 的 SQL 没选 `info_code` → PDF+HTML 双降级全失败。
2. **降级可见化**：降级失败时打印 WARN 日志，不要静默 pass。静默 = 以为正文已提取，实际 LLM 又在读标题。
3. **降级链测试**：新增降级路径后，至少用一个真实 URL 验证每一级都能走到。

> 历史事故：2026-06-14 `_analyze_industry` SQL 查询未选 `info_code` 列，`download_report_pdf` 调用未传 `info_code` 参数，导致 PDF 提取失败后 HTML 降级也失败（因为降级需要 `info_code`），53 行业月报正文覆盖率从应有 80%+ 跌到 54%。

### LLM 输出校验铁律（CRITICAL — 任何 LLM 输出入库/写报告前必读）

**任何 LLM 输出中的股票代码/名称，必须经 `llm_validator` 逐条校验后才能入库或写入报告。** 三道防线：

1. **代码白名单**：`validate_codes()` 校验代码是否真实存在（全市场 5323 只），不存在的标记 rejected。
2. **名称-代码匹配**：`validate_name_code_pairs()` 交叉 API 验证，不匹配的自动修正为真实名称。
3. **行业一致性**：检查标的主概念（`STOCK_PRIMARY_CONCEPT`）是否与催化关键实体有交集，无交集+F EV=0 → rejected。

基础设施统一在 `llm_validator.py`，禁止各文件自造轮子。所有 LLM 管线接入：catalyst_screen / feval Δ / gfactor / unified_scorer / analyze_zsxq / analyze_wechat / primary_synthesis / _run_advice。Dashboard 有 LLM 输出质量行监控幻觉率，>5% 告警。

**概念覆盖率阈值**：`multi_concept_map.json` 中覆盖率 >5% 的概念（标的数/全市场总量）视为噪音，从关键词匹配中排除。该阈值与 防混锅铁律 中的富集比阈值（5%）共用同一逻辑。

> 历史事故：2026-06-17 DeepSeek融资→东风汽车(600006)，根因三重：(1) multi_concept_map 中"DeepSeek概念"含761只标的(14.3%)，(2) LLM 映射 prompt 只发代码没收行业信息，(3) 输出无逐条校验。全项目审计发现 9 个高风险 LLM 调用点零校验。修复：L0 过滤53个噪音概念 + L1 prompt注入标的名称/行业/FEV + L2 逐条校验行业一致性 + L3 Dashboard审计闭环。

### 防混锅铁律（CRITICAL — 任何分类/排名/归因系统变更前必读）

标签不是原因，覆盖率高到一定程度就只是噪音。三道防线：

1. **富集比检验**：标签在全市场覆盖率 >5% 时，它在任何子集（涨停/强势）中的出现不携带信息。必须先算 `ER = P(标签|子集) / P(标签|全市场)`，ER < 0.8 视为噪音。数据源：`CONCEPT_UNIVERSE`（config.py，349 个同花顺标准概念 × 全市场覆盖率）。
2. **单主因归因**：每只标的**只计入一个分类筐**。多标签 = 重复计数 = 噪音标签虚高。归因优先级：同花顺概念板块第一顺位（`STOCK_PRIMARY_CONCEPT`，人工 curated）> 特异性评分（不在标准概念库 + 名称最长）。
3. **独立验证源**：同花顺概念板块第一顺位（plate0611.xlsx Sheet 2）是主题归因的「真相基准」，自动打标签不可直接用于排名。

> 历史事故：2026-06-11 复盘报告 mainline 含「分红除权/摘帽预期/深圳国资/华为合作/房地产」，emerging 含「高送转/回购」，央企(8)/一季报增长(7) 位列主题榜前列——均为噪音。根因：多标签重复计数 + 无富集比检验。修复后全部消失。

### 信息→行动断点模型（CRITICAL — 任何新信息管线设计前必读）

信息的完整价值链：`采集 → 优先级筛选 → 深度分析 → 标的映射 → 走势跟踪 → 交易建议`。当前 ① 采集已覆盖（14 源），②-⑥ 不同程度缺失。每次新建或改造信息管线，按这个断点地图逐格对照。

系统已有两套互补管线覆盖 ②-⑤：
- **catalyst_screen**：Haiku 四维打分（M/S/N/U）→ 机械合成行动性 → Sonnet 交叉验证 → 三级标的映射
- **catalyst_tracker**：催化生命周期状态机 — `active → 每日走势交叉 → price_confirmed(复活提醒) | expired(14天无动静)`

### 催化四维评分 + 机械合成

`行动性 = M×3 + S×2 + N×2 + U×1`（8-80 分）。LLM 做定性判断（幅度大不大/具体不具体/新奇不新奇/紧迫不紧迫），公式做定量合成。机械加分项（多源确认 +5、供给冲击/涨价 +3）独立于 LLM。

模式可复用：**LLM 定性 + 机械定量**适用于任何「从海量信息中挑重点」的场景。

**Δ 减持烈度判断规则**：减持信号必须带比例才能判断烈度。`<0.1% = 噪声(Δ=0)`，`0.1-1% = -1~-2`，`1-3% = -3~-4`，`>3% 或大股东/实控人 = -5~-7`。微额减持不得升级为重大利空。feval.py 已加机械校验：减持+正分→自动 clamp 到 ≤0。

> 历史事故：2026-06-17 炬光科技减持 0.0041%，因公告 content=NULL（PDF 未提取），LLM 只看到标题「减持结果公告」就判 Δ=-5。修复：prompt 加比例判断规则 + 代码层减持信号一致性检查。

### 排名与监控方法论（CRITICAL — 任何双评分/双频率系统设计前必读）

**双轨排名模式 — 不可公度的维度不要强行加权合并**：两个独立的评分体系（如 FEV 基本面质量 vs 催化事件冲击力），强行加权永远有偏。双轨并行：轨A 取前 N1，轨B 取前 N2，去重合并。不是调权重，是分席位。代码：`_build_selection` FEVΔ 7 + 催化行动性 3。

**盘中阈值降级 — 同一信号，timeliness > certainty**：盘中行情只走了一半，不能复用盘后阈值，但等收盘就晚了。同一评分维度 × 盘中/盘后不同阈值。原则：timeliness（及时性）> certainty（确定性），盘中信号强度够就推。

**时间窗口合并 — 昂贵做增量，便宜做存量**：LLM 提取只看新数据（增量），DB 查询覆盖历史窗口（存量）。监控池 = 今日新催化标的 ∪ N 天历史未确认 HIGH/CRITICAL 标的 ∪ 盘中增量扫描。昂贵操作（LLM API）只跑一次，便宜操作（DB 查询 + Redis 行情）覆盖全窗口。

### Dashboard 6 源信号聚合（CRITICAL — 任何信息源权重调整前必读）

`_dashboard.py → _aggregate_signals()` 将 6 个信息源的信号合并为统一排名。每周自动产出，驱动「值得关注的标的」模块。

**6 源及权重**：

| 源 | 单位 | 权重 | 分值范围 | 依据 |
|----|------|:---:|:------:|------|
| 公告 deep_read | score（10~85） | ×1.0 | 10~85 | Stage1 硬筛选后至少值 10 分 |
| 研报 | 篇数 | ×20 | 0~N×20 | 机构专业研究，1 篇即进初步筛选 |
| 新闻边际信号 | 条数 | ×8 | 0~N×8 | Haiku 已做初筛过滤无关内容 |
| 行业深研提及 | 次数 | ×10 | 0~N×10 | 跨机构共识方向，信号质量高 |
| 社交源提及 | 次数 | ×10 | 0~N×10 | 公众号/微博是人工精选信息源 |
| 催化信号 | actionability | ×0.8 | 0~64 | 星球+deep_read 产出，同行动性 |

**交叉源加分**（来源数 = 六个源中有数据的源的数量）：

| 覆盖源数 | 加分 |
|:-------:|:---:|
| 2 | +10 |
| 3 | +20 |
| 4 | +35 |
| 5 | +50 |
| 6 | +65 |

**分级阈值**：

| 级别 | 阈值 | 含义 |
|------|:---:|------|
| 🔬 深度分析 | ≥80 | 多源强信号或单一源极高强度 |
| 👀 初步关注 | 20~79 | 1 篇研报或 2 次社交提及即达标 |

**设计原则**：研报和社交源是重点资源，权重高于公告采集和新闻。通过 Stage1 硬筛选的公告至少值 10 分。所有源不做过滤（低分也纳入），靠复合分排序。

代码：`_aggregate_signals()` + `_hot_themes()`（`_dashboard.py` 662-845 行）。

### 概念父子层级

349 个同花顺概念 → 18 个一级主题（`CONCEPT_HIERARCHY`）。先聚合看方向（大军工 15 次 ER 3.2x），再拆细看矛头（军民融合 4 次 ER 5.6x）。不在映射表中的概念自动保持独立。新增的主题/概念需要更新映射表。

### 迭代方法论：问题 → 量化 → 代码 → 验证 → 固化

1. **发现问题**：从辩论/复盘/实际使用中识别痛点
2. **量化指标**：把模糊感受翻译成可检验的数字（富集比 ER、报告体积 KB、函数耗时 s）
3. **代码落地**：最小改动实现，先跑通再优化（30 行能解决的别写 100 行）
4. **端到端验证**：用真实数据跑一次，对比改前改后
5. **固化规则**：学到的方法论写入 CLAUDE.md（防混锅铁律），下次自动加载

### 协作风格

1. 回答风格：简洁直接，结论先行，必要时附依据。
2. 先规划再执行：非 trivial 任务先出计划，对齐后再动手。
3. 追求简洁：能写 10 行别写 50 行。
4. 精准修改：只改要改的，别顺手优化、别扩大范围。
5. 代码风格：无注释优先，变量名自解释；只在 WHY 不明显时加一行注释。
6. **长会话后主动反思**：累计 ≥5 次工具调用或 ≥3 次 commit 后，主动总结踩坑和发现。分两类：
   - 结构性规则 → 先总结展示 → 用户确认 → 写入 CLAUDE.md
   - 操作备忘录 → 直接追加 LESSONS.md，**写入后告知用户**（git diff 可查细节）

### 五维信息源框架（CRITICAL — 任何新信息源评估前必读）

个股深度研究不是单一维度能覆盖的。五个维度互补，零重叠，各自独立产出价值。

| 维度 | Collector | LLM | 成本/天 | 档案 | 信号类型 |
|------|-----------|:---:|:------:|------|---------|
| ① 公告深研 | `announcement_deep_read` | ✅ Haiku+Sonnet | ~$9 | `deep_read/*.md` | 收购/重组/资产注入/跨界 |
| ② 研报跟踪 | `research_deep_read` | ✅ Sonnet | ~$1 | `research_dossiers/*.md` | 首次覆盖/评级变化/EPS修正 |
| ③ 调研情绪 | `sentiment_track` | ❌ 机械规则 | $0 | 追加到② | 密集/首次/升温 |
| ④ 互动易 | `sentiment_track` | ❌ 关键词 | $0 | 追加到② | 送样/满产/新客户/量产 |
| ⑤ 业绩预告 | `sentiment_track` | ❌ 机械规则 | $0 | 追加到② | 暴增/扭亏/暴雷/修正 |

**设计原则**：有数字的用数字（③④⑤），有文字的才用LLM（①②）。不确定时放行，宁可 LLM 多看几条低分公告也不要在 Python 层误杀。

**猎场分级**：一级（算力硬件+新能源）跑完整 deep_read；二级（机器人+化工材料）仅筛选不跑 LLM。公告侧机会少的领域降级，省 37% 成本。

**数据源选择**：全市场 > universe。公告和研报都已从 universe 过滤改为全市场采集。东方财富 HTTP API（`reportapi.eastmoney.com`）一次 GET 拉全市场研报，比 akshare 逐只调快 100 倍。

**日报**：每天 `daily_collect --tier daily` 自动产出 `Dashboard.md`（五维状态+异常警报+趋势），Obsidian 装 Homepage 插件设为首页。
